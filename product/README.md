# product/ — Backlog & Product Workflow

This directory is the **single source of truth for ongoing product work on
Balance, across both repos** (`balance-backend` and `balance-frontend`). It
exists so that a scheduled agent — or a human — can pick up well-defined work
with fresh context every time.

Why it lives here and not in the frontend: every working session has both
repos checked out, a single backlog avoids sync drift for full-stack features,
and this repo is the system of record (the original story workflow in `todo/`
lived here too). The frontend `CLAUDE.md` points here.

## Layout

| File / dir | Purpose |
|---|---|
| `STATE.md` | Living description of what the app is and contains **today**. First thing to read. |
| `ROUTINE_PROMPT.md` | Operating manual for the scheduled daily agent. |
| `TEMPLATE.md` | Template for new feature specs. |
| `backlog/` | Specs waiting to be built. Filename `NNN-slug.md`; **lowest number = highest priority**; leave gaps of 10 so items can be inserted. |
| `done/` | Completed specs, moved here with completion notes appended. |

## Lifecycle of an item

1. **Idea → spec.** A human drops `NNN-slug.md` into `backlog/` (copy
   `TEMPLATE.md`). Rough is fine — the minimum is a Why and a What; the
   implementer refines the rest and records its interpretation in the PR.
2. **Pick.** The agent takes the lowest-numbered feasible item that is not
   already covered by an open PR (PR bodies carry `Backlog item: <id>`).
3. **Build.** Work lands as one PR per touched repo. The *same* backend PR
   moves the spec `backlog/` → `done/` (appending `## Completion notes`) and
   updates `STATE.md`. Frontend-only items still get a small `docs:` PR here
   for that bookkeeping.
4. **Ship.** The maintainer reviews and merges. release-please accumulates
   merged changes into a release PR per repo; merging *that* publishes
   multi-arch Docker images that production (Raspberry Pi) runs. Nothing
   deploys before the maintainer merges.

## Adding and prioritizing items (human)

- Add: copy `TEMPLATE.md` to `backlog/NNN-slug.md`, commit to `main`.
- Reprioritize: renumber files freely — but avoid renumbering an item that
  already has an open PR (the PR references the old id).
- Cross-repo features are **one** spec with `Scope: full-stack`.
- One spec = one reviewable unit (S or M size). Bigger ideas: split into
  sequenced specs up front, or let the agent split them (`NNNa-…`, `NNNb-…`).

## Relation to other docs

- `todo/` (this repo) — the historical initial build-out: sprints 0–4 are in
  `todo/done/`, and `todo/backlog/sprint-5/` holds unimplemented E2E-test
  stories. New work goes in `product/backlog/`, not there. Sprint-5 stories
  can be promoted into `product/backlog/` when wanted.
- `.claude/thoughts/` (both repos) — engineering research and implementation
  plans (the *how*). `product/` is the *what* and *why*.
- `CHANGELOG.md` (both repos) — generated by release-please. Never hand-edit.
